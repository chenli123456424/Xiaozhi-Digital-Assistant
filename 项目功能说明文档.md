# 小智数码助手 — 项目功能口语化说明文档

> 面向面试场景，用口语化方式解释每个功能模块的实现思路。

---

## 一、整体架构

这个项目是一个 AI 数码助手，前端用 React 写的，后端用 FastAPI，两边通过 WebSocket 保持长连接通信。AI 推理部分用 LangGraph 构建了一条多节点的推理链，搜索用的是 Tavily，大模型用的是阿里云的通义千问，语音合成接的是火山引擎的豆包 TTS，对话记忆存在本地的 SQLite 数据库里。

整体数据流：用户发消息 → 后端从数据库读取记忆生成优化搜索词 → 交给 LangGraph 推理链处理 → 结果实时推送前端渲染 → 对话结束后写入记忆。

---

## 二、对话记忆系统

### 数据库结构

SQLite 表只有四个字段：id、session_id、user_query（用户原始提问）、summary（AI 回答摘要）、created_at（时间）。结构简单，不存冗余信息。

### 写入时机

每轮对话的推理链跑完、AI 回答生成后，立即写入数据库，写完才推送 done 给前端，保证下一轮开始时记忆已经在库里。

写入内容：用户提问直接存原文，不经过任何处理；AI 回答全文交给大模型提炼成不超过 150 字的摘要，要求保留产品名、参数、价格、结论等关键实体，然后存入。

### 三级搜索词生成

每轮对话开始前，记忆服务会根据历史记忆生成优化后的搜索词，分三级处理：

第一级：取最近一条记忆，把上一轮的用户提问和摘要加上当前问题一起给大模型判断，看当前问题是不是上一轮的追问或延续（包括用了指代词"他们/它/这个"，或者是追问细节、区别、对比等）。如果是，生成包含具体主题的完整搜索词。

第二级：如果第一级判断无关，取除最近一条外的所有历史记忆，让大模型判断当前问题是否与某条历史对话相关，相关则生成搜索词。

第三级：两级都无关，返回空字符串，由正常工作流处理。

大模型拒绝时统一输出英文 `UNRELATED`，避免中文变体（"无关"/"不相关"等）导致判断失效。

## 三、LangGraph 多节点推理链

### 整体流程

```
记忆服务生成搜索词
        ↓
classifier → planner → [researcher → synthesizer → critic ↺] → END
                    ↘ general_synthesizer → END
```

### 各节点说明

**Classifier（意图分类）**

如果记忆服务已经生成了搜索词（说明是追问），直接判为 tech，不再调大模型。如果没有记忆关联，把当前问题给大模型判断是 tech 还是 general。

**Planner（搜索词确认）**

记忆服务生成的搜索词已经注入到 state 的 resolved_query 字段，planner 直接读取使用，不再自己调大模型生成搜索词，职责简化为确认和透传。

**Researcher（唯一搜索节点）**

用 resolved_query 做一次 Tavily advanced 深度搜索，最多取 6 个来源。把所有网页内容拼在一起，用一次大模型调用同时完成两件事：生成 2-3 句综合总结（analysis），以及从搜索内容里提取结构化关键参数（items JSON）。key 必须是真实参数名（如产品名、价格、处理器），value 必须是搜索结果中实际出现的具体内容，不能用占位符。

**Synthesizer（专业回答生成）**

把 researcher 的搜索总结、结构化参数、用户原始问题一起喂给大模型，生成 Markdown 格式的专业回答。如果是重试，会带上 critic 的具体反馈告诉模型哪里不对。

**General Synthesizer（通用回答）**

general 路径走这里，直接友好回复，不做搜索，走完直接结束。

**Critic（质量审核）**

把 researcher 提取的结构化参数 JSON 和 synthesizer 生成的回答一起给大模型，让它动态判断回答里的数值和参数是否与搜索结果一致。审核不通过时输出具体反馈，最多重试 2 次。如果搜索结果本身为空（降级场景），直接放行。

## 四、火山引擎 TTS 语音合成

### 多方言支持

维护了一张语言到音色 ID 的映射表，普通话、粤语、闽南语、四川话、东北话各自对应不同音色。前端发消息时带上语言参数，后端查表取对应音色调接口。

### 长文本分段

按句子标点切割，每段控制在 800 字节以内，超长句子强制截断。每段单独合成后拼接成完整 MP3，统一编码成 Base64 返回。合成前先清掉 Markdown 符号，只保留纯文字。

### 前端播放

收到 Base64 音频数据后，用浏览器原生方法解码成二进制，包装成 Blob 生成临时 URL，用 Audio 对象播放。播放结束自动释放 URL 避免内存泄漏。用 React Context 管理全局播放状态，发新消息时自动中断当前播放。

## 五、WebSocket 实时通信

### 为什么用 WebSocket

推理链是多节点的，每个节点完成后都要实时推送进度给前端，HTTP 做不到双向实时通信。

### 消息协议

后端推给前端的消息类型：
- `thought`：researcher 搜索到一个新来源，前端实时展示
- `thought_summary`：搜索综合总结
- `search`：结构化参数数据
- `content_patch`：最终回答的流式文字片段（每次 8 个字符）
- `sources`：参考来源列表
- `retry`：critic 触发重试
- `done`：整条链跑完
- `audio`：TTS 合成的 Base64 音频
- `stopped`：用户主动中断

### Session 管理

前端在 localStorage 里持久化存储 session_id，刷新页面后历史记忆不丢失。

### 停止功能

用户点停止，前端发 stop 消息，后端检测到后中断推理链，推送 stopped 给前端。

## 六、前端界面

### 布局

顶部 Header，左侧侧边栏（数字人 + 语言设置），右侧聊天窗口。侧边栏两个面板用 CSS transform 滑动切换，不是条件渲染，切换流畅。

### 数字人动效

朗读时边框变蓝发光，下方出现音频波形动画，全部 CSS 动画实现。朗读状态通过 React Context 全局共享。

### 消息气泡

AI 回答气泡包含三个可折叠面板：思考过程（实时来源 + 总结）、搜索数据（动态表格，字段根据实际数据决定）、主回答（ReactMarkdown 渲染，支持标题/表格/代码块）。回答渲染前清掉引用角标和参考来源段落，来源由系统单独展示。

### 主题系统

亮色/暗色两套主题，React Context 全局管理，切换不需要刷新。

## 七、技术选型

| 模块 | 技术 | 原因 |
|------|------|------|
| 前端 | React + Vite + Tailwind | 组件化，热更新快，原子化 CSS |
| 后端 | FastAPI | 原生异步，WebSocket 支持好 |
| AI 推理 | LangGraph | 支持有状态多节点图，可条件路由和回环 |
| 大模型 | 通义千问 qwen-turbo | 中文理解好，成本低 |
| 搜索 | Tavily | 专为 AI 设计，返回结构化摘要 |
| TTS | 火山引擎豆包 | 支持多方言，音色自然 |
| 记忆存储 | SQLite | 轻量，无需额外部署 |
| 通信 | WebSocket | 双向实时推送，适合流式输出 |
